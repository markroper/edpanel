PowerSchool API Developer Guide

Version

PowerSchool API version 1.6.0 - PowerTeacher Gradebook API version 2.0.0 - Released December 2014

This edition applies to Release 8.2 of the PowerSchool software and to all subsequent releases and modifications until otherwise indicated in new editions or updates.

Preface

Use the PowerSchool API to develop integrations between your application and PowerSchool. This guide is intended for use by software developers with application and web services development experience.

Note: Use of the PowerSchool API is subject to license agreement with Pearson Education, Inc.. Depending upon your licensing level, some or all of the features described in this guide may be unavailable to you.

Introduction

The PowerSchool API is a set of web services that enable developers to deliver application integrations with PowerSchool.

The PowerSchool API supports all of the features necessary for developers to deliver rich, seamless, excellent quality application integrations with PowerSchool, including:

  • Data Access so that you can read your users' PowerSchool data into your application, write their changes back, and receive data change notifications from PowerSchool.
  • Linking so that your users can easily access your application from PowerSchool and can take advantage of simple to advanced integrated workflows.
  • Single Sign-On so that your users can seamlessly navigate between PowerSchool and your application without reauthentication.
  • Registration so that you can identify PowerSchool instance integrated with your application.

This documentation also includes a set of example applications to help get you started with using the PowerSchool API.

Changes in Version 1.6

Named Query
A Named Query is a new data resource that can be defined by the plugin developer and deployed with the Plugin Package.
All Table Access
All tables in the PowerSchool schema, except for a small list of blacklisted tables, are now accessible using the Named Query and Table Resource.
Access Request
Plugins must now declare an Access Request for all fields that they intend to access using the API.
Student SSN and Lunch Status Security
The student SSN and LUNCHSTATUS fields must be requested in the Access Request to access them through the Student Resource.
Identity Attribute
SAML single sign-on can be enhanced using Identity Attribute to include a variety of user type specific identifiers in the SAML Response.
Extension Fields as SAML Attributes
Extension fields to the STUDENTS, USERS, and GUARDIAN tables can now be sent as SAML Extended Schema Attributes.

Known Issues in Version 1.6

Plugin declared named queries are not available for internal requests
Named Queries that are declared and deployed through a plugin package are only accessible through an external or plugin request. These are requests that include an OAuth access token. They are not accessible through an internal request. This means that plugin defined named queries cannot be invoked through customizations in the PowerSchool portal pages.
Workaround: There is currently no workaround for this issue. Internal access support for named queries is planned for a later release.
Table resource queries to core tables are not available for internal requests
Table Resource external requests support both core and extension tables. But for internal requests, only extension tables can be requested.
Workaround: There is currently no workaround for this issue. Internal access to core tables for the table resource is planned for a later release.
Extended schema declaration and access request validation
When a plugin package contains both extended schema declaration and access request to fields in the new extended schema, the PowerSchool administrator will not be able to enable the plugin because the access request validation is not able to see the new extended schema fields.
Workaround: Create two separate plugins, one that declares the extended schema, another that declares the access request to the fields in the extended schema. The PowerSchool administrator must enable the plugin with the extended schema declaration first, then enable the plugin with the access request.
Invalid extended schema declaration in plugins
Invalid extended schema declarations in plugins, such as those with invalid characters in field and table names, will not generate a visible error message through the plugin installation page and will cause the plugin installation to suspend.
Workaround: The problems with the extended schema definition will be logged in the psj-runtime log. Review the log file and correct the problems, then re-install the plugin.

Versioning Policy

In order to provide assurance to our partners, Pearson uses a versioning policy that defines how we maintain the API. Specifically, there are two kinds of endpoints defined in the API:

  • Versioned endpoints, which have a version number associated with them and which conform to the versioning policy.
  • Unversioned endpoints, which do not have a version number and which do not conform to the versioning policy.

Pearson has defined these two types of endpoints in an effort to have a middle ground that provides backward compatibility guarantees where they are practical, but allows flexibility for certain parts of the API that are thought to be changeable.

The versioning policy is a variation of semantic versioning. Read that document for further information. To summarize the document contents:

  • Version numbers are given as X.Y.Z, where X denotes the major version number, Y denotes the minor version number, and Z is the patch version.
  • Changes that break backward compatibility result in increase of major version number.
  • New functionality that is backward compatibility results in increase of minor version number.
  • Bug fixes only result in increase of patch version number.

In addition to the semantic versioning guarantees, Pearson's versioning policy also attempts (on a "best effort" basis) to make no fewer changes to the API than necessary. That is, a major version change is not seen as an opportunity to change everything.

Since API clients can depend on only backward-compatible changes being made within a major version, only the version number is present on the endpoint URI. For example, /ws/v1/school is used rather than /ws/v1.0.0/school. This is because all sub-versions of 1 are backward compatible and thus should be usable by clients using the same URI.

Finally, the API guide indicates, for each resource and data dictionary element, the first version in which that element was available. Unversioned endpoints will indicate their lack of versioning.
