SAML Single Sign-On

Introduction

SAML single sign-on allows PowerSchool users to seamlessly navigate between PowerSchool and your application without reauthentication.

This document assumes a general knowledge of XML document structure and syntax, and SAML single sign-on technology.

Powerschool Identity Provider needs to be installed on the PowerSchool server in order for SAML SSO to work.

Declare SAML Service Provider Information

Adding a SAML service provider to the PowerSchool user interface is very easy. You simply need to supply a <saml> tag to <plugin>.

Configure Name ID Format

By default the SAML single sign-on plugin will return a Name ID that is constructed from the authenticationId attribute. The authenticationId attribute is defined via the Identity Attribute element in the plugin.

The default authenticationId and Name ID will return a string as described below:

Format PowerSchool URL + "/" + User Context + "/" + User Name
Example http://www.powerschool-district.com/admin/ssmith

However, when the identity attribute element is declared, this Name ID string will be replaced by whatever is configured in that identity attribute.

In addition, the Name ID format can also be configured. In the <saml> declaration, the following Name ID formats may be requested:

Format Description
unspecified Returns the Name ID as the raw value of the authenticationId attribute.
persistent A one-way MD5 hash of the concatenation of the plugin name and the authenticationId attribute.
transient Returns the Name ID as the raw value of the authenticationId attribute.

An example <saml> declaration with the nameid-format is shown below: 

<saml name="MyApp"
    entity-id="http://mycompany.com/myapp"
    base-url="https://mycompany.com"
    metadata-url="http://mycompany.com/path/to/metadata"
    nameid-format="persistent" >

The nameid-format is optional. When it is not specified, the system will use the unspecified format.

Declare Single Sign-On Links

Adding single sign-on links to the PowerSchool user interface is done by including a <links> tag inside the <saml> tag.

This sample XML defines a plugin for MyApp and configures PowerSchool with the information necessary to initiate single sign-on with MyApp. 

<?xml version="1.0" encoding="UTF-8"?>
<plugin xmlns="http://plugin.powerschool.pearson.com"
        xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
        xsi:schemaLocation='http://plugin.powerschool.pearson.com plugin.xsd'
        name="MyApp"
        version="1.0"
        description="Provides SAML single sign-on link to my third-party application">

  <saml name="MyApp"
	    entity-id="http://mycompany.com/myapp"
		base-url="https://mycompany.com" 
		metadata-url="http://mycompany.com/path/to/metadata" >

    <links>
      <link display-text="My App" title="Login to MyApp" path="/sso">
        <ui_contexts>
          <ui_context id="admin.header"/>
          <ui_context id="admin.left_nav"/>
          <ui_context id="teacher.header"/>
          <ui_context id="student.header"/>
        </ui_contexts>
      </link>
    </links>

  </saml>
  
  <publisher name="Example Publisher">
    <contact email="publisher@example.com" />
  </publisher>  
  
</plugin>

Single Sign-On Link Visibility

For an admin or teacher, a plugin SAML single sign-on link will only display if:

  • They are of the user type the link is specified for, and there are no permissions declared in that plugin.
  • There are permissions declared in that plugin, and one or more of the permissions declared in that plugin is turned on for one or more of the user access roles that are assigned to that user.

Declare Single Sign-On Permissions

Adding single sign-on permissions to the PowerSchool user interface is done by including a <permissions> tag inside the <saml> tag.

This sample XML defines a plugin for MyApp and configures PowerSchool with the information necessary to initiate single sign-on with MyApp. 

	<?xml version="1.0" encoding="UTF-8"?>
	<plugin xmlns="http://plugin.powerschool.pearson.com"
			xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
			xsi:schemaLocation='http://plugin.powerschool.pearson.com plugin.xsd'
			name="MyApp"
			version="1.0"
			description="Provides SAML single sign-on link to my third-party application">

	  <saml name="MyApp"
			entity-id="http://mycompany.com/myapp"
			base-url="https://mycompany.com" >
			
		...
		
		<permissions>
			<permission name="P1" description="Permission 1" value="PERM_1" />
			<permission name="P2" description="Permission 2" value="PERM_2" />
			<permission name="P3" description="Permission 3" value="PERM_3" />
		</permissions>
		
	  </saml>
	  
	  <publisher name="Example Publisher">
		<contact email="publisher@example.com" />
	  </publisher>  
	  
	</plugin>

Declare Single Sign-On Attributes

As part of the authentication process the PowerSchool identity provider will include attributes derived from the current user's profile. The plugin author chooses which attributes should be included for each of the four user types allowed: admin, teacher, student, guardian. The attributes for a particular user type need only be specified when links are included with a matching ui context value in the links section of the XML file.

Attributes are added by adding an <attributes> tag with nested <user> tags containing attributes for each user.

Many of the attributes are selected directly from tables in the PowerSchool SIS schema. These are referred to using the PowerSchool schema column name, but can be an alias by using an 'attribute-name' to whatever name the author requires for the attributes in the SAML response.

Certain attributes represent the result of some computation or an extended selection of data from the SIS. Some of these may not be applicable to all user types. In such cases the attribute tag can specify an attribute-value, which will be transmitted in the SAML response.

Student Email Attribute

A special attribute can be defined in the plugin.xml that will cause the federating PowerSchool IdP to include an email address for the current student. Student email is not available as a "built-in" of the STUDENT table; therefore, to include the student email in the authenticated response, the plugin author will have to include a special attribute for this purpose. The attribute name value for this special attribute is "studentEmail" and a typical XML entry would look like this: 

<user type="student">
	<attribute name="FIRST_NAME" />
	<attribute name="LAST_NAME" />
	<attribute name="LASTFIRST" />
	<attribute name="SCHOOLID" />
	<attribute name="studentEmail" />			
</user>

Note: The studentEmail special attribute is only applicable when working within the "student" context.

A typical response would look something like this:

<saml2:Attribute Name="studentEmail">
	<saml2:AttributeValue xmlns:xs="http://www.w3.org/2001/XMLSchema" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" xsi:type="xs:string">student@aghs.edu</saml2:AttributeValue>
</saml2:Attribute>

Institution Attribute

A special attribute can be defined in the plugin.xml that will cause the federating PowerSchool IdP to include a list of valid institutions to which the current user is associated. The plugin.xml attribute is defined for each context, and must indicate the institution identifier to be returned in the single sign-on response.

Depending on the user context, valid institutions will be defined as follows:

Context Table.Column
Student Student.SchoolID
Teacher Teacher.SchoolID
Parent Student.SchoolID for each student associated with this parent.
Admin Teacher.CanChangeSchools

Updating the Plugin

To enable the "institutions" attribute feature, add an attribute with the name "institutions" to your plugin.xml. This attribute must also define the data type to be returned in the single sign-on response. You MUST select one valid data type from the list below, identifying this as an "attribute-value". You may ONLY select one valid data type:

Valid Data Types:

attribute-value Table.Column
id School.ID
dcid School.DCID
name School.NAME
school_number School.SCHOOL_NUMBER
alternate_school_number School.ALT_SCHOOL_NUMBER

A valid institution attribute might look something like this: 

<user type="admin">
	<attribute name="FIRST_NAME" />
	<attribute name="LAST_NAME" />
	<attribute name="LASTFIRST" />
	<attribute name="SCHOOLID" />
	<attribute name="institutions" attribute-value="name" />			
</user>

In the above case, the response XML that is returned from an admin user who has been authorized by the PowerSchool IdP will contain the following: 

<saml2:Attribute Name="institutions">
	<saml2:AttributeValue xmlns:xs="http://www.w3.org/2001/XMLSchema" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" xsi:type="xs:string">District Office</saml2:AttributeValue>
	<saml2:AttributeValue xmlns:xs="http://www.w3.org/2001/XMLSchema" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" xsi:type="xs:string">Washington Elementary</saml2:AttributeValue>
	<saml2:AttributeValue xmlns:xs="http://www.w3.org/2001/XMLSchema" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" xsi:type="xs:string">Apple Grove High School</saml2:AttributeValue>
	<saml2:AttributeValue xmlns:xs="http://www.w3.org/2001/XMLSchema" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" xsi:type="xs:string">Ontario High School</saml2:AttributeValue>
	<saml2:AttributeValue xmlns:xs="http://www.w3.org/2001/XMLSchema" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" xsi:type="xs:string">Cherry Hill Middle School</saml2:AttributeValue>
</saml2:Attribute>

Bear in mind however that any school, which is excluded by the access features of the plugin itself, will not ever be included in any list returned to the service provider application.

Students Attribute

A special attribute can be defined in the plugin.xml that will cause the federating PowerSchool IdP to include a list of valid students to which the current guardian is associated. The plugin.xml attribute is defined for the guardian context only, and must indicate the student identifier to be returned in the single sign-on response.

Updating the Plugin

To enable the "students" attribute feature, add an attribute with the name "students" to your plugin.xml. This attribute must also define the data type to be returned in the single sign-on response. You MUST select one valid data type from the list below, identifying this as an "attribute-value". You may ONLY select one valid data type:

Valid Data Types:

attribute-value Table.Column
id Student.ID
dcid Student.DCID
student_number Student.STUDENT_NUMBER
state_student_number Student.STATE_STUDENTNUMBER

A valid students attribute might look something like this: 

<user type="guardian">
	<attribute name="FIRST_NAME" />
	<attribute name="LAST_NAME" />
	<attribute name="LASTFIRST" />
	<attribute name="SCHOOLID" />
	<attribute name="students" attribute-value="dcid" />			
</user>

In the above case, the response XML that is returned from an admin user who has been authorized by the PowerSchool IdP will contain the following: 

<saml2:Attribute Name="students">
	<saml2:AttributeValue xmlns:xs="http://www.w3.org/2001/XMLSchema" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" xsi:type="xs:string">12345</saml2:AttributeValue>
	<saml2:AttributeValue xmlns:xs="http://www.w3.org/2001/XMLSchema" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" xsi:type="xs:string">12346</saml2:AttributeValue>
	<saml2:AttributeValue xmlns:xs="http://www.w3.org/2001/XMLSchema" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" xsi:type="xs:string">12347</saml2:AttributeValue>
	<saml2:AttributeValue xmlns:xs="http://www.w3.org/2001/XMLSchema" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" xsi:type="xs:string">12348</saml2:AttributeValue>
	<saml2:AttributeValue xmlns:xs="http://www.w3.org/2001/XMLSchema" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" xsi:type="xs:string">12349</saml2:AttributeValue>
</saml2:Attribute>

Extended Schema Attribute

Schema extension fields may also be returned as SAML attributes. The fields must be from 1-to-1 extensions to the USERS, STUDENTS, or GUARDIAN tables.

Declare the attribute within each user type element. For example, the declaration below requests that for students, single sign-on also sends the student's ACT Composite data. 

<user type="students">
	<attribute name="act_composite" attribute-value="act_composite" extension="studentcorefields" />
</user>

The declaration elements are described below:

name Used as the attribute name in the SAML response.
attribute-value The physical field name from the extension table.
extension The physical table name of extension.

In the SAML response the field will be returned as an attribute named act_composite as shown below: 

<saml2:AttributeStatement>
	<saml2:Attribute Name="act_composite">
	     <saml2:AttributeValue xmlns:xs="http://www.w3.org/2001/XMLSchema" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" xsi:type="xs:string">22</saml2:AttributeValue>
	</saml2:Attribute>
</saml2:AttributeStatement>
